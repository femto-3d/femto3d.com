Project Overview

A complete full-stack IoT home automation system demonstrating the entire smart home pipeline: wireless sensor nodes, MQTT message broker, Linux gateway controller, rule engine, and interactive dashboard. The system integrates 5+ distributed sensor nodes (ESP32 & ESP8266 microcontrollers) with a Raspberry Pi central gateway running Python automation scripts.

Each sensor node measures temperature, humidity, motion, gas concentration, and light level — publishing readings every 60 seconds via MQTT. The Raspberry Pi gateway processes sensor data, runs anomaly detection algorithms, triggers actuators (relays for fans, lights, gas cutoff), and logs everything to a local SQLite database. An attractive Node-RED dashboard provides real-time visualization and manual controls.

Key achievement: Sub-100ms latency (average 87ms sensor-to-actuator response). 72-hour continuous operation with zero dropped MQTT messages. Demonstrates production-grade IoT patterns used in industrial automation.

How It Was Built — Step by Step

1

Hardware Layer — Multi-Sensor Nodes

Designed and assembled 5 independent wireless sensor nodes, each consisting of: (1) ESP32 or ESP8266 microcontroller, (2) DHT22 temperature/humidity sensor, (3) HC-SR501 PIR motion detector, (4) MQ-2 gas sensor (detects LPG/smoke), (5) LDR light sensor on ADC, (6) 2-channel relay module for light/fan control. Each node powered by USB 5V adapter or battery pack. All sensor interfaces use standard GPIO/ADC pins. Nodes are housed in 3D-printed enclosures with proper ventilation for gas sensor exposure.

2

MQTT Communication Layer

Installed Mosquitto MQTT broker on Raspberry Pi (Debian-based package). Each sensor node connects to the Pi's local network (192.168.1.x) and publishes sensor readings to structured MQTT topics: home/livingroom/temperature, home/livingroom/humidity, home/kitchen/gas_level, etc. Quality of Service (QoS) set to level 1 for guaranteed delivery with no duplicates. Topic hierarchy allows flexible filtering in downstream subscribers. MQTT retain flag enabled for critical actuator commands (lights, fans) so state persists across broker restarts.

3

Raspberry Pi Gateway — Python Automation Brain

Wrote Python 3 scripts on Raspberry Pi (Raspbian Linux) that subscribe to all MQTT topics using the Paho MQTT library. Main logic flow: (1) Read incoming sensor messages, (2) Apply threshold rules (e.g., if temperature >28°C and humidity >70%, turn on fan relay), (3) Write decision to MQTT actuator topics, (4) Log all sensor readings and actions to local SQLite3 database with timestamp. Implemented multithreaded architecture to prevent MQTT subscriptions from blocking disk I/O. Database schema includes tables for readings (datetime, location, sensor_type, value) and events (timestamp, event_type, action_taken).

4

Node-RED Visual Workflow & Dashboard

Installed Node-RED (JavaScript runtime for visual workflows) on the Raspberry Pi. Built drag-and-drop MQTT subscriber nodes → function nodes → UI dashboard nodes. Dashboard displays: live temperature/humidity gauges, motion detection status, gas sensor readings with color-coded warnings, light sensor graph, and toggle buttons for manual relay control. All dashboard elements update in real-time as sensor data arrives. Dashboard accessible via web browser (http://192.168.1.20:1880/ui) from any device on home network — fully responsive design adapts to mobile screens.

5

OTA Firmware Updates — Zero Downtime Deployments

Implemented Over-The-Air (OTA) firmware update capability using the ArduinoOTA library on all ESP32/ESP8266 nodes. Each node connects to a Python script running on the gateway that serves new firmware binaries. To push an update: (1) compile new MicroPython code on laptop, (2) transfer binary to Raspberry Pi via SCP, (3) trigger OTA update from Node-RED dashboard button, (4) each node downloads and self-flashes in 30 seconds, (5) auto-reboot. Zero downtime — other nodes continue operating while one node updates. Tested firmware rollback by uploading previous version — all node states recovered correctly.

6

Anomaly Detection & Alert System

Developed Python anomaly detection module that maintains a rolling 24-hour window of sensor values for each location. Calculates mean and standard deviation every hour. Any new reading that deviates >2σ (two standard deviations) from the mean is flagged as an anomaly. Examples: gas sensor spike (leak detection), sudden temperature drop (door left open), motion at 3 AM in unoccupied room. Anomalies trigger automatic actions: (1) email alert to homeowner with timestamp and sensor values, (2) if gas above threshold, automatically actuate solenoid valve to disable stove (safety), (3) log event to database with "ALERT" flag. Email dispatch latency <5 seconds (Gmail SMTP via TLS).

7

Performance Validation & Production Testing

Conducted 72-hour continuous operation test without manual intervention. Measured sensor-to-actuator latency using synchronized timestamps: publish sensor reading → log MQTT receive time on gateway → timestamp actuator command → measure time delta. Results: average latency 87ms (min 45ms, max 140ms) on local Wi-Fi. Tested message reliability by intentionally dropping network packets — configured QoS 1 prevented any lost messages over 72 hours. Monitored database size: average growth 50MB per week at 60-second sampling interval across 5 nodes. Implemented automatic database compression (SQLite VACUUM) to maintain performance. Zero crashes, zero missed messages, 100% uptime.
